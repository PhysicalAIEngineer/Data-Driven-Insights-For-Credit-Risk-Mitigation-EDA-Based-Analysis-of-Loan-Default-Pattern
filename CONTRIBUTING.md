# 🤝 Contributing Guidelines

Thank you for your interest in contributing to this project! 🎉
We welcome contributions from everyone and appreciate your effort to improve this project.

---

## 📌 How to Contribute

You can contribute in several ways:

* 🐛 Reporting bugs
* 💡 Suggesting new features
* 📝 Improving documentation
* 📊 Adding new analysis or visualizations
* ⚡ Optimizing code or performance

---

## 🚀 Getting Started

### 1. Fork the Repository

Click the **Fork** button at the top right of this repository.

---

### 2. Clone Your Fork

```bash
git clone https://github.com/<your-username>/credit-risk-analysis-eda.git
cd credit-risk-analysis-eda
```

---

### 3. Create a New Branch

```bash
git checkout -b feature/your-feature-name
```

---

## 🛠️ Development Setup

### 🔹 Install Dependencies

```bash
pip install -r requirements.txt
```

---

### 🔹 Run Jupyter Notebook

```bash
jupyter notebook
```

---

## 📊 Contribution Types

### 🔹 1. Code Contributions

* Follow clean and readable coding practices
* Add comments wherever necessary
* Ensure your code runs without errors

---

### 🔹 2. EDA Enhancements

* Add meaningful visualizations
* Provide business insights with plots
* Avoid redundant or duplicate analysis

---

### 🔹 3. Documentation Improvements

* Improve README clarity
* Add explanations to notebooks
* Fix grammar or formatting issues

---

## ✅ Contribution Guidelines

Before submitting a pull request:

* ✔️ Ensure your code is properly tested
* ✔️ Follow consistent naming conventions
* ✔️ Keep commits small and meaningful
* ✔️ Write clear commit messages

Example:

```bash
git commit -m "Added default rate analysis by income group"
```

---

## 🔄 Pull Request Process

1. Push your changes:

```bash
git push origin feature/your-feature-name
```

2. Open a Pull Request (PR)

3. Provide:

   * Description of changes
   * Why the change is needed
   * Screenshots (if applicable)

---

## 📏 Code Style Guidelines

* Use **PEP 8** for Python code
* Use meaningful variable names
* Keep functions modular and reusable
* Avoid unnecessary complexity

---

## 🧪 Testing

* Ensure notebooks run end-to-end without errors
* Verify visualizations are generated correctly
* Validate results before submission

---

## 🚫 What Not to Do

* ❌ Do not submit broken or incomplete code
* ❌ Avoid large, unrelated changes in one PR
* ❌ Do not duplicate existing work

---

## 💬 Communication

If you have questions:

* Open an **Issue**
* Start a discussion
* Ask for clarification before implementing major changes

---

## 🏆 Recognition

All contributors will be acknowledged for their work.
We value your time and effort in improving this project! 🙌
